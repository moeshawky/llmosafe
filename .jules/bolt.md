## 2024-04-14 - Optimizing Redundant String Manipulations in Sifter
**Learning:** `get_bias_breakdown` utilized an array-based sliding window (`[&str; 4]`) inside a loop over words. On *every* iteration, it populated the array, sliced the first 3 words, iterated over them, and applied `trim_matches()` to check for negations. This led to O(K) redundant work. `calculate_utility` compared every word in an `observation` to every word in an `objective` inside a nested loop, repeatedly applying `trim_matches()` on the `objective` words. This caused O(N*M) redundant string operations.
**Action:** Replaced the array-based sliding window with a simple integer Time-To-Live (`ttl`) counter (`negation_ttl`). This reduces the negation tracking to O(1) state management while flawlessly preserving the logic. Replaced the nested loop trimming by caching up to 64 trimmed `objective` words on the stack in a `[""; 64]` array. This is perfect for `no_std` environments, avoiding heap allocations while maintaining high performance for typical inputs, and falling back gracefully for larger inputs.
## 2025-04-15 - [Hoist invariant array building in sift_perceptions]
**Learning:** In `sift_perceptions`, calling `calculate_utility(obs, objective)` inside a loop over observations unnecessarily repeats the parsing and caching of the `objective` string into a fixed-size `[&str; 64]` array on every iteration. This redundant string parsing is a significant bottleneck in a `#![no_std]` codebase processing untrusted data.
**Action:** Always inspect loops containing functions that build localized caches or parse invariant data. Extract the cache-building logic to run exactly once before the loop, and pass the populated stack array to an inner helper function by reference.
## 2025-05-16 - [Cache derived string properties to avoid redundant post-loop computation]
**Learning:** In `sift_perceptions`, `calculate_halo_signal(obs)` was correctly computed per observation to find the optimal string, but then `calculate_halo_signal(best_obs)` was inexplicably run *again* on the chosen string outside the loop. Derived string properties for the selected max/min element should be cached alongside the reference string during the search, especially in `no_std` context where string matching overhead adds up.
**Action:** When finding a "best" element in a loop, if any derived property computed *during* the search is needed *after* the search, cache it directly in a `best_property` variable alongside the `best_element` pointer.
